수행평가 폭탄주의보 프로젝트 보고서 초안

1 주제 선정 이유

고등학생은 여러 과목의 수행평가를 동시에 준비해야 함
수행평가는 마감일, 반영 비율, 과목별 난이도, 개인이 느끼는 스트레스가 모두 다르기 때문에 단순히 날짜만 보고 관리하면 우선순위를 정하기 어려움
특히 같은 주에 여러 수행평가가 몰리면 어떤 과제부터 시작해야 하는지 판단하기 힘들고 준비 시간이 부족해지는 문제가 생김

이 프로젝트는 수행평가 정보를 등록하고 이번 주에 몰린 과제 수를 분석하며 마감일과 반영 비율과 스트레스 점수를 바탕으로 우선순위를 추천하는 백엔드 시스템을 만드는 것을 목표로 함
학생이 수행평가 일정을 더 체계적으로 관리하고 과제 폭탄 상황을 미리 파악할 수 있도록 돕는 것이 핵심 목적임

2 이론적 배경 및 선행 서비스 분석

기존 일정 관리 앱은 날짜와 알림 중심으로 작동함
하지만 수행평가는 단순 일정과 다르게 과목, 반영 비율, 예상 소요 시간, 개인 스트레스 같은 학습 관련 요소가 중요함
일반 캘린더 앱은 특정 과목에서 스트레스가 반복적으로 높게 나타나는지, 이번 주 과제가 위험 수준인지, 어떤 과제를 먼저 시작해야 하는지까지 분석해 주지는 않음

본 프로젝트는 일반 일정 관리 기능에 수행평가 특화 분석 기능을 더했다는 점에서 차별점이 있음
과목 테이블과 수행평가 테이블을 ORM 관계로 연결하고 수행평가 데이터를 기반으로 대시보드 통계, 우선순위 추천, 과목별 피드백 요약을 제공함
추후에는 기록된 피드백 데이터를 바탕으로 간단한 머신러닝 모델을 연결하여 예상 스트레스 점수를 예측하는 기능까지 확장할 수 있음

3 시스템 설계

3-1 FastAPI 디렉토리 구조

Project/
  database py
  models py
  schemas py
  service py
  main py
  seed py
  assessments db

database py는 데이터베이스 연결과 세션 생성을 담당함
models py는 SQLAlchemy ORM 모델을 정의함
schemas py는 요청과 응답에 사용하는 Pydantic 모델을 정의함
service py는 실제 비즈니스 로직을 담당함
main py는 FastAPI 엔드포인트를 정의하고 service py의 함수를 호출함
seed py는 테스트용 초기 데이터를 생성함

이 구조는 API 계층과 데이터 처리 로직을 분리하기 때문에 유지보수가 쉽고 오류가 발생했을 때 어느 계층에서 문제가 생겼는지 확인하기 쉬움

3-2 데이터베이스 모델

Subject 테이블
  id 과목 고유 번호
  name 과목명
  category 과목 분류
  credit_hours 이수 단위

PerformanceAssessment 테이블
  id 수행평가 고유 번호
  subject_id 과목 id
  evaluation_type 수행평가 유형
  weight 반영 비율
  due_date 마감일
  overlapping_tasks 같은 주에 겹치는 과제 수
  actual_time_spent 실제 소요 시간
  user_stress_score 사용자가 기록한 스트레스 점수

Subject와 PerformanceAssessment는 1:N 관계임
하나의 과목은 여러 개의 수행평가를 가질 수 있고 하나의 수행평가는 하나의 과목에 속함
SQLAlchemy의 relationship과 ForeignKey를 사용하여 두 테이블을 연결함
이 구조는 ORM 관계 조건을 만족하며 과목별 피드백 분석 기능에서 두 테이블의 관계를 활용함

3-3 ERD 설명

Subject 1 ---- N PerformanceAssessment

Subject는 과목 정보를 저장하는 부모 테이블임
PerformanceAssessment는 각 과목에 속한 수행평가 정보를 저장하는 자식 테이블임
PerformanceAssessment의 subject_id가 Subject의 id를 참조함

3-4 유스케이스 다이어그램 설명

사용자는 과목을 등록할 수 있음
사용자는 수행평가를 등록할 수 있음
사용자는 수행평가 완료 후 실제 소요 시간과 스트레스 점수를 기록할 수 있음
사용자는 대시보드에서 이번 주 수행평가 폭탄 여부를 확인할 수 있음
사용자는 우선순위 추천 목록을 확인할 수 있음
사용자는 과목별 스트레스 리포트를 확인할 수 있음

4 구현 결과

4-1 핵심 API

GET /subjects
등록된 과목 목록을 조회함

POST /subjects
새 과목을 등록함

GET /assessments
등록된 수행평가 목록을 조회함

POST /assessments
새 수행평가를 등록함
등록 전에 subject_id에 해당하는 과목이 존재하는지 확인함

POST /assessments/{assessment_id}/feedback
수행평가 완료 후 실제 소요 시간과 스트레스 점수를 기록함

GET /dashboard/summary
현재 날짜부터 7일 이내의 수행평가 개수를 계산함
수행평가 개수에 따라 위험, 경고, 보통 상태를 반환함
for 반복문과 리스트 집계, if-elif-else 조건문을 활용한 대시보드 통계 기능임

GET /assessments/priority
마감일까지 남은 일수와 반영 비율을 조합하여 우선순위 점수를 계산함
sorted 함수를 활용해 우선순위 점수가 높은 순서대로 수행평가를 정렬함

GET /analytics/user-report
과목별로 사용자가 기록한 스트레스 점수를 모아 평균을 계산함
과목과 수행평가의 ORM 관계를 활용하여 개인화된 피드백 리포트를 제공함

4-2 핵심 기능 옵션 구현 설명

옵션 1 수행평가 폭탄 주의보
이번 주에 마감되는 수행평가 개수를 계산하고 개수가 4개 이상이면 위험, 2개 이상이면 경고, 그 외에는 보통으로 판단함
학생이 과제 폭탄 상황을 미리 파악할 수 있도록 돕는 기능임

옵션 2 우선순위 해결사
마감일이 가까울수록 반영 비율이 높을수록 우선순위 점수가 높아지도록 계산함
계산된 점수로 수행평가 목록을 정렬하여 지금 먼저 시작해야 할 과제를 추천함
추후에는 사용자의 스트레스 점수까지 가중치에 포함하여 더 개인화된 추천을 만들 수 있음

옵션 3 나만의 스트레스 분석가
사용자가 기록한 피드백 데이터를 과목별로 모아 평균 스트레스 점수를 계산함
이를 통해 어떤 과목에서 스트레스가 높게 나타나는지 확인할 수 있음

5 머신러닝 연동 계획

채점 기준에서 ML 연동이 필수이므로 현재 구조에 다음 방식으로 간단한 머신러닝 기능을 추가할 계획임

입력 데이터
  weight
  overlapping_tasks
  actual_time_spent

예측 대상
  user_stress_score

구현 방식
  수행평가 피드백 데이터가 충분히 쌓이면 weight, overlapping_tasks, actual_time_spent를 입력값으로 사용하고 user_stress_score를 예측값으로 사용하는 간단한 모델을 만듦
  수업 범위를 크게 벗어나지 않도록 처음에는 규칙 기반 예측 함수를 만들고 필요하면 간단한 회귀 모델을 연결함

예상 엔드포인트
  POST /analytics/predict-stress

기대 효과
  사용자가 수행평가 정보를 입력하면 예상 스트레스 점수를 미리 확인할 수 있음
  이를 통해 어떤 과제를 먼저 준비해야 하는지 판단하는 데 도움을 줄 수 있음

6 트러블 슈팅

6-1 문제 1 ML 라이브러리 사용 범위 문제

문제 상황
초기 코드에는 pandas, scikit-learn, joblib을 사용한 머신러닝 파일이 있었음
하지만 수업 범위가 ch01부터 ch09까지로 제한되어 있었기 때문에 외부 ML 라이브러리 사용이 제출 조건과 충돌할 가능성이 있었음

원인 분석
기능 요구사항에는 ML 연동이 있었지만 수업 범위 제한 조건도 동시에 존재했음
따라서 무거운 외부 라이브러리를 그대로 사용하는 방식은 감점 위험이 있었음

해결 과정
우선 핵심 FastAPI, ORM, 관계형 테이블 구조를 안정적으로 정리하기 위해 ML 관련 파일과 모델 파일을 제거함
이후 service py 중심으로 수행평가 폭탄주의보, 우선순위 추천, 스트레스 리포트 기능을 구현함
최종 제출 전에는 채점 기준의 ML 필수 조건을 만족하기 위해 간단한 규칙 기반 예측 또는 가벼운 모델 연동 방식으로 다시 추가할 계획임

배운 점
기능을 많이 넣는 것보다 채점 기준과 수업 범위를 동시에 만족하는 설계가 중요하다는 것을 알게 됨
요구사항이 충돌할 때는 우선순위를 정하고 감점 위험이 큰 부분을 먼저 정리해야 함

6-2 문제 2 Python 캐시 권한 오류

문제 상황
python3 -m py_compile 명령으로 문법 검사를 실행했을 때 macOS Python이 홈 디렉토리의 Library/Caches에 캐시 파일을 만들려고 하면서 PermissionError가 발생함

원인 분석
코드 문법 오류가 아니라 Python이 pycache 파일을 생성하는 위치에 권한이 없어서 발생한 문제였음

해결 과정
PYTHONPYCACHEPREFIX 환경 변수를 사용하여 캐시 생성 위치를 /tmp/pycache-check로 변경함
이후 같은 컴파일 검사를 다시 실행했고 정상적으로 통과함

배운 점
에러 메시지를 볼 때 코드 오류인지 실행 환경 문제인지 구분해야 함
PermissionError는 코드 로직보다 파일 시스템 권한 문제일 가능성이 높음

7 생성형 AI 활용 내역

활용 도구
Codex 기반 AI 개발 도우미

활용 목적
FastAPI 프로젝트 구조 설계, ch01부터 ch09 범위 기준 코드 검수, 불필요한 ML 코드 제거, service 계층 분리, 보고서 초안 작성, 트러블 슈팅 정리

핵심 프롬프트 예시
  Project 디렉토리 안에 있는 모든 코드가 ch01~ch09 내용만 사용하는지 검수해줘
  필요없는 부분은 지우고 ch01~ch09와 비슷한 구조로 다시 설계해서 코딩해줘
  모든 주석을 삭제해줘
  한국어나 print문 안의 설명을 적을 때 다 한국어로 해줘
  수행평가 채점 기준과 최종 제출물 요구사항을 바탕으로 보고서 내용을 생성해줘

AI 사용에 대한 윤리적 설명
생성형 AI는 프로젝트를 대신 완성하는 도구가 아니라 설계 방향을 점검하고 코드 구조를 개선하는 보조 도구로 사용함
AI가 제안한 내용은 직접 검토하고 수업 범위와 채점 기준에 맞게 수정함

8 배운 점 및 느낀 점

이번 프로젝트를 통해 FastAPI로 API를 만들고 SQLAlchemy ORM으로 테이블 관계를 구성하는 방법을 익힘
특히 단순히 데이터를 저장하는 것에서 끝나는 것이 아니라 저장된 데이터를 분석하여 사용자에게 의미 있는 정보를 제공하는 과정이 중요하다는 것을 배움

또한 프로젝트를 진행하면서 기능 요구사항, 수업 범위, 채점 기준을 모두 고려해야 한다는 점을 느낌
처음에는 ML 기능을 크게 구현하려고 했지만 수업 범위와 제출 조건을 검토하면서 더 안정적인 구조로 정리하는 과정이 필요했음
앞으로는 기능을 구현하기 전에 요구사항을 먼저 분석하고 감점 위험이 없는 구조를 설계하는 습관을 가져야겠다고 느낌
